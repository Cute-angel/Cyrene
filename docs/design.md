# Coding Agent 记忆系统设计

## 1. 文档目的

本文定义一套供多个 Coding Agent 共用的程序性记忆系统。系统从用户要求和已经验证成功的任务轨迹中提炼可复用的操作方式，并在后续任务进入相应场景时自动为 Agent 注入少量、高相关的记忆。

本文描述产品行为、模块边界、运行形态、记忆模型、匹配算法和索引边界。本文确定 Core 与 Installer 分别构建为独立可执行程序，但暂不确定编程语言、数据库、向量模型或 Agent SDK。

## 2. 产品定位

### 2.1 核心目标

系统需要实现以下能力：

1. 多个 Coding Agent 共用同一套个人记忆。
2. 记忆能够跨项目复用，不依赖某个仓库的对话历史。
3. 用户可以通过 `/remember` 从当前任务片段中提炼记忆。
4. 获得写入授权的 Agent 可以从已经验证成功的任务中自动新增记忆。
5. 系统在任务开始和具体操作开始前匹配相关记忆。
6. 自动注入以高精度为优先，宁可漏掉，也不应大量误注入。
7. 用户可以查看、纠正、停用、删除和整理已有记忆。

### 2.2 记忆的含义

本系统保存的是“以后遇到类似任务时应该怎样操作”，即程序性记忆。

适合保存的内容：

- 修改现有内容前先分析影响范围。
- 只修改请求涉及的文件和行为。
- 生成 Word 文档时不要使用 pandoc。
- 编辑压缩包容器格式的文档时，先备份，再替换目标内部文件，最后重新打包。

不适合保存的内容：

- 这次把登录页改成蓝色。
- 当前项目的后端端口是 8080。
- 上次某个命令失败了。
- 因为之前排版出错，所以不要使用某个工具。
- Agent 的内部推理过程、完整工具输出或原始对话。

失败、纠正和验证结果可以作为生成记忆的临时证据，但最终记忆正文只能包含操作规则或操作步骤。

### 2.3 范围边界

第一阶段只保存跨项目可复用的个人操作习惯和通用流程，不承担项目知识库职责。

以下内容应由项目自己的说明文件管理，而不是写入全局记忆：

- 项目专用构建命令。
- 仓库目录结构。
- 业务规则和接口地址。
- 只对单一项目有效的格式约定。
- 临时任务参数和一次性需求。

## 3. 系统架构

系统按职责拆分为 `core`、`adapter` 和 `installer` 三部分。核心原则是：通用能力集中在 `core`，Coding Agent 的差异封装在 `adapter`，安装环境的变更集中由 `installer` 管理。

```text
Coding Agent
    ↕
对应的 Adapter
    ↕ 稳定的核心协议
Core
    ↕
记忆存储与检索索引

Installer ── 安装、升级、修复或卸载 Core 与 Adapter
```

### 3.1 Core

`core` 是系统唯一的核心逻辑实现，负责：

- 记忆的创建、读取、搜索和治理。
- 任务级与操作级匹配。
- 结构化、全文和向量索引的统一调用。
- 候选召回、排名融合和最终注入校验。
- 权限、指令优先级、冲突关系和负反馈处理。
- 对 Adapter 提供稳定、与具体 Agent 无关的调用协议。

`core` 不负责识别某个 Coding Agent 的私有配置格式、命令注册方式或会话数据结构，也不直接修改 Agent 的安装目录。

`core` 编译为独立可执行程序，建议产物命名为 `cyrene-core`。它应能够脱离源码目录直接运行，不要求用户额外启动语言解释器或开发工具链。记忆数据、配置和日志保存在可执行文件之外，以便升级程序时保留用户数据。

`core` 可以采用常驻服务或按需启动的运行方式，具体进程生命周期留到实现阶段确定；无论采用哪种方式，对 Adapter 暴露的协议都应保持一致。

### 3.2 Adapter

`adapter` 是 Coding Agent 与 `core` 之间的接入层。每个受支持的 Coding Agent 都有自己的 Adapter，例如：

```text
adapter/
  codex/
  claude-code/
  gemini-cli/
```

每个 Adapter 只负责该 Agent 特有的工作：

- 注册 `/remember` 和记忆管理命令，或映射为该 Agent 支持的等价入口。
- 将该 Agent 可观察到的任务片段、操作阶段和成功证据转换为核心协议。
- 调用 `core` 获取候选记忆，并把核心返回的记忆原文注入该 Agent。
- 将用户确认、撤销和“本轮不适用”等操作传回 `core`。
- 适配该 Agent 的插件、Hook、MCP、Skill、配置文件或其他扩展机制。

Adapter 不实现记忆存储、匹配算法、权限判断或最终注入限制，避免不同 Agent 各自复制核心逻辑并产生行为差异。

Adapter 不要求统一编译为独立程序。它可以根据目标 Agent 的扩展方式表现为插件、脚本、配置包或薄客户端，但必须只通过公开核心协议访问 `core`，不得直接读写核心数据库。

### 3.3 Installer

`installer` 负责所有安装状态变更，包括：

- 安装、升级、修复和卸载 `core`。
- 检测本机已安装且受支持的 Coding Agent。
- 按用户选择安装或卸载各 Agent 的 Adapter。
- 写入、更新和清理 Adapter 所需的 Agent 配置。
- 初始化数据目录和基础配置，并在卸载时明确区分程序文件与用户记忆数据。
- 执行安装前检查、版本兼容性检查和安装后连通性验证。

`installer` 不承担记忆匹配或日常运行逻辑，也不应成为 Adapter 调用 `core` 的中转层。

`installer` 编译为另一个独立可执行程序，建议产物命名为 `cyrene-installer`。它应能够脱离源码目录直接运行，不要求目标机器预装语言解释器或开发工具链。安装与卸载应由同一程序提供，既可以暴露交互式流程，也可以提供可自动化的命令行参数。

### 3.4 依赖方向

模块依赖必须保持单向：

```text
Adapter  ──调用──> Core 公开协议
Installer ──管理──> Core 与 Adapter 的安装状态
Core ──不依赖──> 任何具体 Adapter 或 Installer
```

核心协议应进行版本化。Adapter 在调用前声明自身协议版本，`core` 返回兼容结果或明确的升级要求。Installer 根据兼容矩阵选择能够配套工作的 Core 与 Adapter 版本，不能通过让 `core` 引入 Agent 专用代码来解决兼容问题。

### 3.5 运行流程

日常任务中的调用链如下：

1. Coding Agent 的 Adapter 从当前任务中形成标准请求。
2. Adapter 调用正在运行或按需启动的 `core`。
3. `core` 完成过滤、召回、校验和数据读写。
4. Adapter 将核心结果转换为该 Agent 支持的命令响应或上下文注入。
5. 用户反馈和任务结果通过 Adapter 回传给 `core`。

安装和卸载不经过上述运行链路，而是由用户直接启动 `cyrene-installer` 完成。卸载 Adapter 不应删除记忆数据；卸载 Core 时，Installer 必须让用户明确选择保留或删除个人数据。

### 3.6 源码与构建产物

建议源码边界与当前目录保持一致：

```text
src/
  core/           核心逻辑与核心程序入口
  adapter/        各 Coding Agent 的独立适配实现
  installer/      安装器逻辑与安装器程序入口
```

发布时至少生成两个彼此独立的可执行程序：

```text
cyrene-core
cyrene-installer
```

不同平台可以使用相应文件名，例如 Windows 下的 `cyrene-core.exe` 和 `cyrene-installer.exe`。Adapter 则按各 Agent 所需格式分别打包，并由 Installer 统一部署。是否将 Adapter 资源嵌入 Installer、随安装包附带或按需下载，留到发布设计阶段决定。

## 4. 记忆模型

### 4.1 记忆类型

系统支持两种记忆正文。

#### 规则（Rule）

规则是可以独立执行的简短操作约束。

```text
修改现有内容前先分析影响范围。
```

每条规则只表达一个主要操作要求。复合要求应在生成候选时拆分，以避免部分匹配时引入无关约束。

#### 流程（Procedure）

流程是有固定顺序的操作步骤。

```text
1. 创建备份。
2. 解包目标文件。
3. 只替换需要修改的内部文件。
4. 重新打包。
```

流程正文只保存步骤，不保存触发原因、失败经过、推理说明或验收解释。适用条件由系统作为隐藏索引维护。

同一种操作可以保存多条独立流程。新流程不会自动覆盖旧流程，多条流程通过名称和隐藏适用条件区分。

### 4.2 用户可见字段

用户直接查看和编辑的内容包括：

- 短名称：由 Agent 自动生成，用户可以修改。
- 类型：规则或流程。
- 正文：规则文本或步骤列表。
- 状态：启用、停用或归档。
- 来源：用户或 Agent 名称。

### 4.3 系统隐藏字段

以下信息用于搜索、匹配和治理，默认不要求用户填写或查看：

- 唯一标识。
- 受控分类标签。
- 开放关键词。
- 专用检索文本。
- 语义向量及其版本。
- 创建时间和更新时间。
- 写入来源类型。
- 来源 Agent 名称。
- 重复、冲突和备选流程关系。
- 场景级负反馈。
- 索引版本和正文版本。

隐藏字段不得包含原始对话、Agent 自由推理、失败原因或完整任务记录。

### 4.4 来源记录

来源分为两类：

- 用户明确要求。
- Agent 从成功任务轨迹中总结。

Agent 来源只记录 Agent 名称，不记录模型版本、会话标识、对话摘要或推理过程。

## 5. 记忆生成

### 5.1 任务片段

`/remember` 不只读取用户最后一条消息，而是分析当前完整任务片段。任务片段可以包含：

- 用户最初的任务要求。
- 用户后续追加的约束和纠正。
- Agent 明确展示的判断和行动。
- 工具执行结果。
- 发生过的返工和最终成功路径。
- 测试、构建、文件校验等客观结果。

系统不能依赖或要求 Agent 暴露未公开的内部思维链，只能使用任务中可观察的信息。

### 5.2 用户主动生成

用户执行 `/remember` 后，系统执行以下流程：

1. 收集当前任务片段。
2. 提取用户明确表达的长期操作要求。
3. 从已经成功的操作路径中提取可复用步骤。
4. 排除项目事实、一次性参数和未经验证的推断。
5. 将候选拆分成规则或流程。
6. 删除原因、历史描述和对话引用。
7. 检测重复和冲突。
8. 展示候选名称、正文和来源类型。
9. 用户逐条保存、编辑或丢弃。
10. 用户确认后写入并建立检索索引。

用户主动生成仍然采用预览确认，不因 Agent 获得自动写入权而省略。

### 5.3 Agent 自动生成

获得用户授权的 Agent 可以在任务完成后自动新增记忆，但必须同时满足：

- 任务存在客观验证结果，或者用户明确确认任务成功。
- 经验能够跨项目复用。
- 不是一次性任务内容。
- 不是对已经注入记忆的简单重复执行。
- 与现有记忆不存在未处理的冲突或近似重复。
- 当前任务尚未达到自动写入上限。

每个成功任务最多自动新增一条记忆。自动写入后立即显示记忆名称和来源，并允许用户撤销、编辑或停用。

仅靠 Agent 自己声称“已完成”不构成自动写入的成功证据。没有测试、构建、文件校验或用户确认时，Agent 只能轻量提示“本次任务可能存在可记忆经验”。

### 5.4 后续纠正

如果 Agent 刚自动写入记忆，用户随后指出本任务仍有错误：

1. 自动停用本任务刚写入的记忆。
2. 不直接修改该记忆正文。
3. 继续完成用户纠正后的任务。
4. 修正后的任务再次获得成功证据时，可以重新生成一条新记忆。

### 5.5 Agent 写入权限

- 新接入的 Agent 默认只有读取权。
- 用户可以单独授予某个 Agent 自动写入权。
- Agent 只能新增记忆。
- Agent 不能直接修改、合并、停用或删除已有记忆。
- Agent 可以提出上述维护建议，但必须由用户确认。

## 6. 指令优先级与注入行为

### 6.1 指令优先级

所有场景采用固定优先级：

```text
当前用户要求 > 项目自身规则 > 全局记忆
```

记忆只是默认操作方式，不能覆盖用户本轮明确要求，也不能破坏项目已经声明的构建、格式或协作约定。

发生冲突时，本轮按更高优先级指令执行，并在任务结束时建议用户判断该冲突是一次例外，还是需要调整长期记忆。

### 6.2 两级触发

记忆匹配发生在两个阶段：

#### 任务级匹配

在理解任务后、制定行动方案前，匹配适用范围较广的规则，例如：

- 先分析影响范围。
- 保持改动范围最小。
- 保留现有行为和视觉效果。

#### 操作级匹配

在进入某种具体操作前，匹配对应流程，例如：

- 编辑 XMind 文件。
- 生成 DOCX 文档。
- 执行数据库迁移。
- 修改 Git 历史。

一个任务从分析、编码转向文档生成时，可以在新的操作阶段重新检索，但不应在每个细小工具调用前重复注入。

### 6.3 注入限制

- 每次最多自动注入三条记忆。
- 其中最多包含一条流程。
- 多条流程同时相关时，只选择最匹配的一条。
- 无法明确区分多条流程时，不自动选择，将它们列为可能相关。
- 注入用户确认过的原文，不针对当前任务改写或重新概括。
- 不向 Agent 注入隐藏标签、搜索分数或生成原因。

### 6.4 用户提示

自动应用记忆时，默认显示名称摘要：

```text
已应用 2 条记忆：最小改动范围、压缩包文档编辑流程
```

用户可以展开查看完整正文，也可以将某条记忆标记为“本轮不适用”。

## 7. 匹配与搜索总体设计

匹配采用“硬过滤、混合召回、排名融合、当前 Agent 精排、核心校验”的流程。

```text
当前任务或操作
    ↓
生成标准检索描述
    ↓
硬过滤
    ↓
结构化索引 + 全文索引 + 向量索引
    ↓
排名融合并取前 6 条
    ↓
当前 Agent 选择
    ↓
核心限制与冲突校验
    ↓
原文注入最多 3 条，其中最多 1 条流程
```

向量相似度只负责发现候选，不能单独决定自动注入。

## 8. 查询描述

系统不应直接使用整段对话进行搜索，而应先将当前任务或操作归一化为简短检索描述。

```json
{
  "stage": "operation",
  "action": "edit",
  "object": "xmind_package",
  "task_type": "document_editing",
  "environment": ["windows", "powershell"],
  "tools": ["xmind"],
  "keywords": ["标注", "思维导图", "xmind"],
  "explicit_constraints": ["修改现有文件", "保留其他内容"]
}
```

任务级查询描述可以比较宽泛，操作级查询描述必须包含具体动作和操作对象。查询描述只表达“当前准备做什么”，不记录用户执行任务的原因。

## 9. 索引设计

### 9.1 受控分类索引

受控分类保证不同 Agent 使用统一概念，避免同义标签造成索引碎片。

初始分类可以包含：

```text
action:
  inspect | create | edit | debug | refactor | test | build | run
  generate | convert | migrate | review | deploy | delete

object:
  code | config | dependency | database | document | archive_document
  spreadsheet | presentation | image | ui | git_history | environment

task_type:
  analysis | code_change | diagnosis | document_editing | generation
  testing | migration | review | deployment
```

受控词表应保持小而稳定。Agent 可以提出分类，但最终必须归一化到统一值。

### 9.2 开放关键词索引

具体工具、框架、文件格式和新概念进入开放关键词，例如：

```text
xmind, docx, rust, powershell, react, sqlite, 备份, 解包
```

无法归入受控分类的内容只能作为开放关键词，不应自动扩展核心分类词表。

### 9.3 全文索引

全文索引覆盖：

- 记忆名称。
- 规则正文。
- 流程步骤。
- 开放关键词。
- 专用检索文本。

全文检索负责匹配文件格式、工具名称、命令、框架名和用户原始用词。对于中文，应支持中文词语或字符片段检索；对于代码场景，应保留标识符、扩展名和命令名称。

### 9.4 专用检索文本

流程正文只有步骤，可能缺少适用对象。因此系统需要为搜索生成一段不可见的 `retrieval_text`：

```text
编辑 XMind 或其他压缩包容器文档；备份、解包、替换指定内部文件、重新打包。
```

`retrieval_text` 可以包含适用动作和对象，但不能包含失败历史、原因、原始对话或 Agent 推理。

### 9.5 向量索引

向量索引基于记忆名称、正文和 `retrieval_text` 建立，用于处理：

- 同义表达。
- 中英文混合查询。
- 用户没有使用记忆原始关键词的场景。

向量及其模型版本属于可重建索引，不是记忆本体。更换向量模型时应重建向量索引，不改变已确认的记忆正文。

### 9.6 关系索引

记忆之间可以建立以下关系：

- `duplicate`：表达基本相同，建议合并。
- `conflict`：在相同场景给出不兼容要求。
- `alternative`：同一操作的不同可行流程。

冲突记忆不能同时自动注入。备选流程只能选择其中最匹配的一条。

### 9.7 负反馈索引

负反馈必须绑定场景，而不是直接降低整条记忆的全局权重。

```json
{
  "memory_id": "mem_123",
  "action": "edit",
  "object": "react_component",
  "environment": ["typescript"],
  "feedback": "not_applicable"
}
```

场景级负反馈由所有 Agent 共用，同时记录当时由哪个 Agent 完成了选择，以便后续分析。负反馈不保存 Agent 的选择理由。

## 10. 自动匹配算法

### 10.1 硬过滤

候选召回前先排除：

- 已停用或已归档的记忆。
- 当前 Agent 无权读取的记忆。
- 明确违反当前用户要求的记忆。
- 明确违反项目规则的记忆。
- 命中相同场景负反馈的记忆。
- 对操作对象有硬性要求但当前对象不匹配的流程。

### 10.2 三路召回

每次搜索同时执行：

1. 结构化召回：根据动作、对象、任务类型、环境和工具匹配。
2. 全文召回：根据名称、正文、关键词和检索文本匹配。
3. 语义召回：根据查询描述与记忆向量相似度匹配。

每一路可以先返回 Top 20～30，再合并去重。

### 10.3 排名融合

不同索引的原始分数不可直接相加。系统应基于各路结果排名进行融合，初始优先级为：

```text
结构化匹配 > 全文匹配 > 语义匹配
```

可以采用以下概念性权重作为起点：

```text
recall_score =
    2.0 × structured_rank_score
  + 1.2 × lexical_rank_score
  + 1.0 × semantic_rank_score
  - negative_feedback_penalty
```

这些权重只用于确定 Agent 能看到哪些候选，不能直接决定注入。后续根据真实匹配样例调整，不将某个固定余弦相似度当作通用阈值。

### 10.4 候选预算

排名融合后最多向当前 Agent 提供六条候选。六条候选应包含：

- 候选 ID。
- 类型。
- 名称。
- 已确认的原始正文。

不提供原始对话、失败原因、自由推理或其他 Agent 的选择解释。

候选数量较小，因此召回阶段应优先优化相关记忆是否进入前六名，而不是扩大最终注入数量。

### 10.5 当前 Agent 精排

当前 Agent 根据完整任务上下文选择适用记忆。输入示例：

```json
{
  "operation": {
    "action": "edit",
    "object": "xmind_package",
    "environment": ["windows"]
  },
  "candidates": [
    {
      "id": "mem_a",
      "type": "rule",
      "content": "修改前先分析影响范围。"
    },
    {
      "id": "mem_b",
      "type": "procedure",
      "content": [
        "创建备份。",
        "解包目标文件。",
        "替换指定内部文件。",
        "重新打包。"
      ]
    }
  ]
}
```

Agent 只返回被选中的记忆 ID：

```json
{
  "selected": ["mem_a", "mem_b"]
}
```

Agent 不返回匹配说明，也不能修改、拼接或重新生成记忆正文。

不同 Agent 可能对相同候选作出不同选择。系统通过统一候选协议、候选预算、核心硬限制和共享负反馈控制差异，而不是要求所有 Agent 的语义判断完全一致。

### 10.6 核心最终校验

Agent 返回选择后，核心再次验证：

- 选择项必须来自本次候选集合。
- 总数不得超过三条。
- 流程不得超过一条。
- 冲突记忆不得同时出现。
- 备选流程不得同时出现。
- 更高优先级指令覆盖的记忆不得注入。
- 记忆状态在查询期间发生变化时，以最新状态为准。

校验通过后，系统只注入记忆原文。

## 11. 手动搜索

手动搜索与自动匹配共用索引，但采用不同目标：

- 自动匹配以高精度为优先，最多向 Agent 提供六条候选。
- 手动搜索以找全为优先，放宽匹配门槛并返回更多结果。

手动搜索支持：

- 自然语言查询。
- 精确关键词查询。
- 按规则或流程筛选。
- 按来源 Agent 筛选。
- 按启用、停用或归档状态筛选。
- 查看相似、冲突和备选记忆。

低置信结果可以出现在手动搜索中，但不能因此获得自动注入资格。

## 12. 反馈和长期维护

### 12.1 本轮不适用

用户将一条记忆标记为“本轮不适用”后：

1. 本轮停止使用该记忆。
2. 记录动作、对象和环境组合。
3. 所有 Agent 在相似场景中对该记忆降权或排除。
4. 不影响该记忆在其他场景中的使用。
5. 重复出现误匹配后，建议用户调整隐藏适用范围或停用记忆。

### 12.2 防止自我强化

Agent 按已注入记忆完成任务，不能把相同行为再次总结成新记忆，也不能仅因任务成功自动提高该记忆的权重。

只有以下信息可以形成新的有效反馈：

- 用户明确提出的新要求。
- 用户明确标记“有用”或“不适用”。
- 与已有记忆不同且经过验证的新成功路径。
- 用户确认的合并、编辑或范围调整。

### 12.3 重复和冲突

- 近似重复：展示合并后的建议版本，由用户确认是否替换。
- 明确冲突：不自动覆盖，由用户选择替换旧记忆或调整适用范围。
- 多种有效流程：分别保存为备选流程，不强制合并或覆盖。

### 12.4 归档和删除

- 长期未使用或经常误匹配的记忆只生成归档建议。
- 系统不自动删除记忆。
- 用户可以手动停用、归档或永久删除。

## 13. 安全与隐私边界

### 13.1 写入来源安全

- 用户可以通过 `/remember` 主动触发候选生成。
- 获得授权的 Agent 可以按既定条件自动新增。
- 仓库 README、代码注释、网页内容、工具输出和其他第三方内容不能自行授权长期写入。
- 项目内容中的“请记住这条规则”只能作为不可信输入，不能替代用户授权或成功证据。

### 13.2 权限安全

记忆不能用于：

- 授权删除或覆盖文件。
- 绕过 Agent 原有审批。
- 自动发布、部署或发送外部消息。
- 扩大网络、文件系统或账户权限。
- 覆盖当前用户要求或项目安全规则。

### 13.3 数据最小化

长期存储只保留执行规则、操作步骤和必要索引信息。以下内容必须在候选生成后丢弃或不进入记忆系统：

- 原始对话全文。
- Agent 内部推理。
- 完整终端输出。
- 项目源代码。
- 密钥和令牌。
- 失败原因和历史叙述。

## 14. 管理命令语义

具体命令形式可以由不同 Agent 适配，但产品层应支持以下动作：

```text
/remember            从当前任务片段生成候选记忆
/memory used         查看本轮应用的记忆
/memory search ...   搜索已有记忆
/memory edit         编辑记忆
/memory disable      停用记忆
/memory delete       删除记忆
/memory review       查看重复、冲突、误匹配和归档建议
```

不同 Agent 可以使用不同命令语法，但必须映射到一致的核心语义。

## 15. 完整示例

### 15.1 任务执行

用户要求：

```text
在现有 XMind 文件中增加标注，其他内容不要修改。
```

任务执行中，Agent 采用并验证了以下成功路径：

```text
备份文件 → 解包 → 只替换目标内部文件 → 重新打包
```

### 15.2 生成候选

用户执行 `/remember` 后得到：

```text
[用户明确要求｜规则]
名称：保持最小改动范围
正文：修改现有内容时，只修改请求涉及的部分。

[Agent 总结｜流程]
名称：压缩包文档局部编辑
正文：
1. 创建备份。
2. 解包目标文件。
3. 只替换需要修改的内部文件。
4. 重新打包。
```

系统不会保存 Agent 之前尝试过什么、为什么失败或完整工具输出。

### 15.3 后续匹配

另一个项目中的任务：

```text
在这个 XMind 文件里标记考试重点，不要改变原有结构。
```

标准查询描述：

```text
action=edit
object=xmind_package
task_type=document_editing
constraints=preserve_unrelated_content
```

召回候选：

```text
A 保持最小改动范围
B 压缩包文档局部编辑
C 修改代码后运行单元测试
```

当前 Agent 选择 A 和 B。核心排除 C，并校验结果包含一条规则和一条流程，没有冲突。

最终提示：

```text
已应用 2 条记忆：保持最小改动范围、压缩包文档局部编辑
```

最终注入的是 A 和 B 的已确认原文。

## 16. 总体验收场景

完整路线暂不设置固定误注入率指标，以下场景覆盖当前 v1 与后续阶段；Adapter v1 的实际验收范围以 `docs/adapter-design.md` 为准：

1. 用户明确要求能够通过 `/remember` 生成规则候选。
2. 成功任务能够生成流程，失败或结果未知的任务不能自动生成 Agent 经验。
3. 一次性需求和项目事实不会进入全局记忆。
4. Agent 自动写入每个任务最多一条，并记录 Agent 名称。
5. 用户随后纠正任务时，刚自动写入的记忆会被停用。
6. 当前用户要求和项目规则能够覆盖全局记忆。
7. 结构化、全文和语义召回能合并为最多六条候选。
8. 当前 Agent 只能从候选 ID 中选择，不能修改正文。
9. 最终注入不超过三条且最多包含一条流程。
10. 冲突和备选流程不会被同时注入。
11. “本轮不适用”只影响相似场景，并由所有 Agent 共用。
12. 手动搜索能够找到低置信记忆，而低置信结果不会自动注入。
13. 已注入记忆不会因为被执行而自动强化或重复生成。
14. 仓库内容和第三方输入不能自行授权写入长期记忆。

## 17. 后续待定事项

### 17.1 Core 基础实现已确定事项

Core 第一版基础实现已经确定以下技术边界：

- 使用 Rust 构建独立的 `cyrene-core` 可执行程序。
- 使用 bundled SQLite 保存记忆、Actor、结构化分类、FTS5 索引和向量。
- 同时支持版本化 JSON Lines stdio 与本机回环 HTTP 协议。
- HTTP 使用 Bearer Token，Token 只保存 Argon2 哈希；Agent 默认只读，授权后可以新增，但不能治理已有记忆。
- 全文检索使用 FTS5 trigram；语义检索使用可配置的 OpenAI-compatible embeddings API、模型和向量维度。
- Embedding 服务不可用时先保留本地写入，将向量标记为待索引，并降级为结构化与全文检索。
- 首版提供记忆 CRUD、手动搜索、最多六条自动候选、Actor 管理、索引重建、短期 match session 和 ID-only 最终注入硬校验；Agent 精排由 Adapter 完成，关系治理留给后续阶段。

公开协议与运行方式见 `docs/core-protocol.md`。

### 17.2 Adapter v1 已确定事项与剩余待定

Adapter v1 已确定以下事项：

- 首批支持 Codex 与 pi-agent，分别使用 Codex plugin 和 pi-agent TypeScript package。
- 任务级匹配由 Agent 生命周期事件自动触发，操作级匹配由 Agent 在进入新的具体操作阶段时主动触发。
- Adapter/Agent 负责候选语义筛选，Core 负责候选归属、数量、流程数、状态和版本等硬校验。
- Adapter 按需启动 Core；进程单实例与并发启动由 Core 负责。
- v1 提供主动记忆和管理入口，但暂不实现成功任务自动写入、关系治理与 `memory review`。
- Windows 和 Linux 是 Adapter v1 的验收平台，Token 保存到用户专属受限权限配置文件。

详细协议和接入结构见 `docs/adapter-design.md`。

以下内容仍留到后续阶段决定：

- Adapter 资源随 Installer 内置、随安装包附带还是按需下载。
- 受控分类词表的版本管理方式。
- 查询、注入和反馈事件的保留周期。
- 基于真实样例的召回权重和质量指标。
